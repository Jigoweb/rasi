3 approcci architetturali

La configurazione vive sull'emittente e si applica automaticamente all'import. Differisce per dove agisce e quanto è granulare.

A

Config flat per emittente

Colonna configurazione_import JSONB sulla tabella emittenti. Contiene tutte le regole di normalizzazione + parametri matching. L'import le legge e applica. UI semplice: form strutturato nella scheda emittente.

Cosa configura l'admin:
  • Normalizzazione titolo (maiuscolo → title case, strip "(Original)", strip articoli)
  • Mapping tipo (tv→serie, movie→film, live→live)
  • Campo data (data_trasmissione vs sales_month come proxy)
  • Righe da escludere (es. tipo=live)
  • Tolleranza anno matching (±N anni senza penalità)
  • Soglia minima punteggio matching

Pro

  • Semplice da costruire
  • Config sempre visibile sulla scheda emittente
  • Nessuna migrazione complessa

Contro

  • Tutti gli upload dello stesso emittente usano stessa config
  • Nessun override al volo
B

Config emittente + override per campagna ⭐ Raccomandato

Config salvata sull'emittente (come A), ma ogni campagne_programmazione può sovrascrivere parametri specifici al momento del caricamento. Preview delle prime 10 righe trasformate prima di confermare.

Config emittente (persistente):
  • Normalizzazioni titolo
  • Mapping tipi
  • Tolleranza anno default
  • Filtri riga default
Override campagna (per upload):
  • Tolleranza anno specifica
  • Filtro tipo diverso
  • Soglia matching
Preview smart: upload → sistema applica config emittente → mostra 10 righe trasformate → admin conferma o aggiusta → carica

Pro

  • Veloce per emittenti noti (config già pronta)
  • Flessibile per variazioni di formato
  • Preview impedisce errori silenziosi

Contro

  • Più complesso da costruire (wizard + config)
  • Due posti dove vive la config
C

Pipeline a stadi con validazione interattiva

L'import diventa un wizard multi-step: upload → rilevamento automatico problemi → review admin (anomalie evidenziate) → trasformazione → carica. Nessuna config salvata: ogni upload è guidato.

1. Upload → 2. Auto-detect anomalie → 3. Review problemi → 4. Trasforma → 5. Carica
Auto-detect: % titoli uppercase, % righe senza tipo, % episodi con numero anomalo, % date mancanti. Admin sceglie come risolvere ogni classe di problema.

Pro

  • Massima visibilità sui problemi
  • Non richiede config pre-impostata
  • Robusto a file inattesi

Contro

  • Upload diventa lungo anche per file puliti
  • Config non riutilizzabile → ripete lavoro ogni volta
  • Più costoso da implementare
💡 Nota:

In tutti e 3 gli approcci, la tolleranza anno diventa un parametro configurabile (default ±2 anni senza penalità, ±5 con penalità ridotta) invece di essere hardcoded nella funzione SQL. Questo vale per il matching indipendentemente dall'approccio scelto.
